Анти-паттерны
=============

Чего избегать при использовании CloudCastle DI.

Service locator в домене
------------------------

**Плохо:** ``Container`` или ``ContainerRegistry::get()`` внутри бизнес-классов.

**Лучше:** composition root и зависимости через конструктор.

Недоверенные идентификаторы
---------------------------

Не используйте пользовательский ввод как id сервиса без белого списка.

Глобальный реестр
-----------------

``ContainerRegistry`` — для bootstrap и legacy, не для каждого класса домена. ``reset()`` в тестах.

Слепой ``scan()``
-----------------

Фильтруйте namespace; не сканируйте весь ``src/`` и ``vendor/`` без необходимости.

Autowiring
----------

Attributes, intersection-типы и autowiring по имени параметра **поддерживаются**, как и autowiring **свойств** и **методов** (``enablePropertyAutowiring()``, ``enableMethodAutowiring()``), но не заменяют явный composition root.

Compiled container и YAML/XML — **не** поддерживаются.

Циклы в фабриках
----------------

Autowiring обнаруживает циклы; ``set()`` с фабриками — нет.

Сериализация контейнера
-----------------------

Не сериализуйте контейнер с замыканиями. Собирайте граф при старте процесса.
